Agent
思考过程
先说结论：fm-cli 是单进程 CLI Agent，没有用户体系、没有跨用户匹配、没有持久化多租户。这个课题的难点不在"写一个 Agent"，而在多用户状态 + 多 Agent 协作 + 跨用户匹配 + 可解释。所以 fm-cli 的 Agent 内核（main loop / orchestrator / blackboard / tools / tracer / tokenBudget / antiloop / structured-output）能直接复用，但外围要重写。
下面是企业级规划。
________________________________________
1. 整体思路
一句话：把 fm-cli 的"一个 Agent 跑一个用户任务"改造成"orchestrator 编排三个 worker Agent，共享一份用户画像黑板，跨用户做向量召回 + 规则排序"。
三 Agent 不是简单串行流水线（那样匹配延迟会很高），而是事件驱动 + 阶段切换：
黑板是核心：三个 Agent 不直接互相调用，全靠读写黑板解耦。这样某个 Agent 挂了/慢了不影响别的，便于独立评测和替换。

3. 三 Agent 协作架构
3.1 Orchestrator（编排者，单例）
职责：维护用户会话状态机，决定下一阶段交给哪个 Agent。
状态机（每个用户一份）：
状态转换由信号触发，不是写死的函数调用。信号来自黑板（如 profile.confidence > 0.7 → 触发匹配）。这样可异步、可重放、可测试。
3.2 ProfileAgent（画像采集）
输入：用户最近 N 轮对话 + 当前 profile 草稿 输出：增量 profile patch（不是整份重写，避免抖动）
Profile 结构（这是核心数据契约，定死）：
关键设计：
•	增量 patch：每轮只产 diff，黑板上 merge，version +1。避免重写抖动。
•	confidence 必填：每个字段单独打分。interests[0].confidence 低就不进匹配加权。
•	evidence 锚点：每条结论附"用户原话 + 第几轮"，匹配后给用户看"为什么觉得你喜欢 X"——这是课题的"可解释推荐"亮点。
•	embedding 文本拼接：用一段固定模板把 profile 渲染成自然语言再向量化，比直接向量化字段值召回准。
3.3 MatchAgent（匹配决策）
两阶段：召回 → 排序。延迟敏感（<2s），所以召回走向量库（不调 LLM），排序走规则引擎（也不调 LLM）。LLM 只在"可解释生成"阶段介入。
召回（Recall）
向量距离用余弦（<=>）。pgvector 一行 SQL，零额外服务。
排序（Rank）
混合打分，权重写到配置表（不要硬编码，运营要调）：
每条候选输出 factor 拆分，前端可视化成雷达图。这是"可解释"的关键。
可解释
每个最终候选给一段 LLM 生成的"为什么你们可能合得来"，输入是双方 profile + 评分因子，强制结构化输出：
3.4 IceBreakerAgent（撮合辅助）
输入：双方 profile + 可解释因子 输出：3 条破冰话术 + 使用建议（哪条更适合同步聊 vs 异步聊）
约束：
•	不暴露对方隐私（不引用具体对话原话，只引用 profile 字段）
•	多样性：3 条分别走"共同点 / 互补点 / 场景化"三个角度
•	风格可控：根据双方 socialStyle.depth 调整深浅

8. 几个关键决策（面试可能问）
1.	为什么黑板模式而不是流水线？
流水线下游必须等上游，画像没稳定就没法匹配。黑板允许 ProfileAgent 持续微调画像的同时，MatchAgent 用上一版画像召回——解耦延迟。
2.	为什么 pgvector 而不是 Milvus/Pinecone？
用户量 <100 万时 pgvector 完全够，少一个服务少一份运维。等到百万级再迁移，向量库接口已经在 packages/vector-store/ 抽好了。
3.	为什么画像要增量 patch？
整份重写会丢失"用户上一秒说自己外向，下一秒改口"的演进信息。evidence 锚点 + version 让画像可追溯。
4.	为什么匹配不调 LLM？
LLM 排序延迟 1-3s 且贵。规则引擎 50ms，可解释用 LLM 在选完 top-3 后再调一次，成本降 90%。
5.	多 Agent 怎么防死循环？
直接搬 fm-cli 的 antiloop.ts：每个 Agent 内部记签名计数，跨 Agent 用黑板锁防止两个 Agent 互相触发。
